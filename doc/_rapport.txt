# Les grandes lignes du projet
## Analyse de la situation initiale
La compréhension d’un projet existant représente un défi récurrent dans le métier de développeur, particulièrement lorsque le code est volumineux, complexe ou mal documenté. Avant de pouvoir contribuer à un projet, un développeur doit explorer manuellement les fichiers, identifier les dépendances et reconstituer mentalement la logique générale de l’application. C’est une phase qui peut prendre beaucoup de temps. La correction et la détection de bug dans une application peut aussi être une tâche longue et compliqué. Même avec une grande attention, on passe parfois à coté de bug ou de solution non optimisé.
Ce projet vise justement à faciliter, non seulement la compréhension du code, via des schémas et des explications, mais aussi à détecter et proposer des solutions à des erreurs présent dans le code.

## Technologies
Décrire chaque domaine pertinent et les variantes possibles
### Framework CSS
Tailwind CSS :
Tailwind est un framework CSS dit utility-first. Il met à disposition une classe pour chaque propriété de style (p-4 pour le padding, text-blue-500 pour la couleur, etc.) qui s'applique directement dans le HTML. Cette approche évite l'écriture d'une feuille de style séparée et offre un contrôle total sur le design. Le framework est entièrement personnalisable via un fichier de configuration permettant de définir ses propres couleurs, espacements et points de rupture.
Bootstrap :
Bootstrap propose une approche orientée composants. Il fournit des blocs préfabriqués prêts à l'emploi (boutons, cartes, modales, barre de navigation, formulaires, etc.) que l'on applique via des classes regroupant plusieurs propriétés CSS. Cette approche accélère le démarrage d'un projet, mais impose un style visuel reconnaissable, plus difficile à personnaliser sans écrire des surcharges.
### Langage backend
JavaScript Node.js :
Node.js est un environnement d’exécution JavaScript côté serveur, basé sur le moteur V8 de Google Chrome. Il fonctionne sur un modèle asynchrone non bloquant ce qui est adapté aux applications faisant de nombreux appels à des services externes. Il dispose d’un vaste écosystème de paquets disponible via NPM et permet aussi d’utiliser le même langage que le frontend Vue.js.
PHP :
PHP est un langage interprété conçu spécialement pour le développement web.
Java Springboot :
Java c’est un langage fortement typé, très répandu dans le monde de l’entreprise. Spring Boot est un framework qui simplifie le développement d’applications Java en fournissant une configuration par défaut et un écosystème riche en modules.
### Modèle d’IA (API LLM)
OpenRouter :
OpenRouter est une passerelle qui donne accès, via une seule API compatible avec celle d’OpenAI, à de nombreux modèles de langage provenant de différents fournisseurs (OpenAI, Anthropic, Google, etc.). Le modèle à utiliser se choisit par une simple variable de configuration, sans toucher au code, et un plafond de crédit peut être fixé sur le compte pour maîtriser les dépenses.
API d’un fournisseur unique (par exemple OpenAI) :
Il est aussi possible d’appeler directement l’API d’un seul fournisseur, comme l’API d’OpenAI citée en exemple dans le cahier des charges. Cette approche est simple et bien documentée, mais elle lie le projet à ce fournisseur et à ses seuls modèles.

## Sécurité de l’information et protection des données
Conformité nLPD, mesures techniques et organisationnelles
L’application web sera sécurisé afin d’éviter les attaques par injection (html, css, js et sql). Les requêtes seront protégées via l’API afin d’éviter des accès non autorisés, seuls les utilisateurs authentifiés pourront accéder aux données et uniquement à leurs propres données.
Du point de vue de la nouvelle loi sur la protection des données (nLPD), l’application ne collecte aucune donnée personnelle. Les seules informations enregistrées sont l’identifiant GitHub de l’utilisateur et son token d’accès, utilisés uniquement pour la connexion et les appels à GitHub, le token étant chiffré en base. Aucune donnée n’est utilisée à d’autres fins. De plus, dans le cadre du TPI, le projet n’est pas hébergé et reste en usage local, il n’y a donc pas de risque réel sur ce point.
## Diagramme de cas d’utilisation
Diagramme de cas d’utilisation de CodeMentor. Le diagramme ci-dessus représente l’ensemble des cas d’utilisation de l’application CodeMentor et les acteurs qui les déclenchent. Deux acteurs humains apparaissent : le « Visiteur », qui correspond à toute personne arrivant sur l’application sans être authentifiée, et l’« Utilisateur », qui hérite du Visiteur une fois la connexion via GitHub OAuth réalisée. Trois acteurs externes complètent le système : l’API GitHub pour l’authentification et la récupération du code, l’API OpenRouter pour l’analyse par intelligence artificielle, et la base de données interne pour la persistance des projets et de leurs analyses. Les cas d’utilisation principaux concernent la gestion des projets, le lancement et la consultation d’analyses, ainsi que la connexion et la déconnexion.
Détail des cas d’utilisation. Chaque cas correspond à une fonctionnalité accessible depuis l’interface :
Se connecter. Le Visiteur s’authentifie via OAuth GitHub depuis la page de login. Après validation par GitHub, une session est créée et le Visiteur devient Utilisateur.
Se déconnecter. L’Utilisateur clôt sa session active. Les cookies sont invalidés et il revient à l’état Visiteur.
Ajouter un projet. L’Utilisateur fournit l’URL d’un dépôt GitHub. L’application vérifie auprès de l’API GitHub que le dépôt existe et que l’Utilisateur y a accès, puis crée le projet dans la base.
Consulter la liste des projets. L’Utilisateur voit le tableau de bord avec tous ses projets liés à GitHub. Un filtre permet de rechercher par nom ou statut d’analyse.
Consulter un projet. L’Utilisateur ouvre la page d’un projet pour voir ses métadonnées (langage principal, visibilité, date) et l’historique de ses analyses.
Supprimer un projet. L’Utilisateur supprime un de ses projets. L’opération est bloquée si une analyse est en cours, et les analyses associées sont supprimées en cascade.
Lancer une analyse. L’Utilisateur déclenche l’analyse d’un projet. Le backend crée une tâche en file d’attente, qui sera traitée de manière asynchrone : récupération du code via l’API GitHub, analyse fichier par fichier via l’API OpenRouter, puis stockage des résultats.
Consulter les résultats d’analyse. L’Utilisateur consulte le rapport d’une analyse terminée : synthèse globale, diagramme structurel Mermaid, liste des fichiers et erreurs détectées avec leurs corrections proposées.
## Diagramme de séquences système
Diagramme de séquences, création d’une analyse. L’utilisateur lance une analyse depuis l’interface. Le backend enregistre l’analyse en BDD avec le statut « queued » et la met en file d’attente, puis répond immédiatement au client. Le traitement effectif est détaillé dans le diagramme ci-dessous.
Diagramme de séquences, traitement asynchrone d’une analyse. Le worker traite les analyses en arrière-plan, en boucle sur la file. Il récupère le code via l’API GitHub, le passe à OpenRouter pour analyse, et met à jour l’avancement en BDD à chaque étape. Ce découplage évite de bloquer la requête HTTP pendant les minutes que peut prendre un appel IA.
Diagramme de séquences, consultation de l’état d’avancement. Pendant qu’une analyse tourne, le client interroge son état toutes les 2 secondes (polling). Le serveur lit le pourcentage et le statut en BDD, et les renvoie. C’est ce qui alimente la barre de progression de l’écran « Analyse en cours ».
## Diagrammes d’activités
Diagramme d’activités, ajouter un projet. L’utilisateur colle l’URL d’un dépôt GitHub. Deux vérifications enchaînées : existence du dépôt, puis permissions de lecture. Chaque échec déclenche un message d’erreur dédié (introuvable / accès refusé). Si tout est OK, le projet est créé et l’utilisateur est redirigé vers sa page.
Diagramme d’activités, consulter une analyse. L’utilisateur ouvre une analyse depuis la page d’un projet. Si elle est encore en cours, on bascule sur l’écran d’avancement (cf. polling plus haut). Sinon, il accède aux résultats consolidés : explications du code, erreurs/fixes, et diagramme structurel Mermaid.
Diagramme d’activités, supprimer un projet. Deux garde-fous : on bloque la suppression si une analyse tourne (évite les tâches orphelines dans le worker), et on demande une confirmation explicite avant d’agir. Les analyses associées sont supprimées en cascade.

La section « Historique des analyses » récapitule dans un tableau toutes les analyses lancées sur le projet avec leur identifiant, leur date, leur statut (succès, échec) et le nombre d’erreurs détectées. Les boutons « Voir sur GitHub » et « Lancer une analyse » permettent respectivement d’ouvrir le dépôt en ligne ou de déclencher une nouvelle analyse, tandis que « Consulter » réouvre les résultats d’une analyse passée.

## Concept d‘implémentation
### Conception de l’API REST
L’API REST de CodeMentor expose les fonctionnalités de l’application sous forme d’endpoints HTTP consommés par le frontend Vue.js. Toutes les réponses sont au format JSON et la documentation interactive de l’API est générée avec Swagger, accessible sur l’URL /codementor/api-docs.
L’authentification repose sur une session HTTP-only ouverte après le flux OAuth GitHub. Toutes les routes en dehors de /auth/* requièrent une session valide et retournent un code 401 sinon. Une ressource qui appartient à un autre utilisateur est traitée comme inexistante (code 404) pour éviter toute fuite d’information.
Authentification :
GET /auth/login : redirige vers le flux OAuth GitHub.
GET /auth/callback : reçoit le code GitHub, crée ou met à jour l’utilisateur et ouvre la session.
POST /auth/logout : termine la session (code 204).
GET /auth/me : retourne l’utilisateur connecté de la session (le token GitHub est exclu de la réponse).
Projets :
GET /projects : liste les projets de l’utilisateur.
POST /projects : ajoute un projet à partir d’une URL GitHub. Renvoie 201 si tout va bien, 404 si le dépôt est introuvable, 403 si l’utilisateur n’a pas accès au dépôt, 409 si le projet existe déjà pour cet utilisateur.
GET /projects/:id : détail d’un projet, de ses métadonnées GitHub et de son historique d’analyses.
DELETE /projects/:id : supprime le projet et son historique d’analyses. Renvoie 409 si une analyse est en cours sur ce projet, conformément au diagramme d’activité de la suppression.
Analyses :
POST /projects/:id/analyses : lance une nouvelle analyse. L’API enregistre l’analyse en base avec le statut queued et renvoie immédiatement le code 202 avec l’identifiant de l’analyse. Le traitement effectif est délégué au worker asynchrone.
GET /analyses/:id : résultat complet d’une analyse terminée : vue d’ensemble, fichiers analysés, erreurs détectées avec leurs corrections suggérées et diagramme Mermaid.
GET /analyses/:id/status : endpoint léger interrogé en boucle par le frontend pour afficher la barre de progression. Il renvoie le statut courant et le pourcentage d’avancement.
Les codes HTTP utilisés suivent les conventions REST : 200 pour une lecture, 201 pour une création, 202 pour une création asynchrone, 204 pour une suppression, 400 pour un payload invalide, 401 pour une session manquante, 404 pour une ressource introuvable et 409 pour un conflit (doublon ou analyse en cours).
## Modèle relationnel de la base de données
Le schéma relationnel ci-dessous présente la base de données CodeMentor sous forme de diagramme en patte d’oie. Le schéma compte cinq tables reliées par des clés étrangères avec suppression en cascade. La suppression d’un utilisateur entraîne la suppression de ses projets, de leurs analyses, des fichiers analysés et des erreurs/corrections associées.
Le diagramme a été produit avec MySQL Workbench.
### Table t_user
Représente un utilisateur authentifié via OAuth GitHub. Aucun mot de passe n’est stocké. Le système délègue l’identification à GitHub et conserve uniquement le token OAuth nécessaire aux appels ultérieurs.
[TABLEAU 5x4]
  | Champ | Type | Contraintes | Rôle et justification
  | pk_user | INT | PK, AUTO_INCREMENT | Identifiant interne.
  | github_id | INT UNSIGNED | NOT NULL, UNIQUE | Identifiant numérique GitHub, immuable même si l’utilisateur change de login. UNSIGNED car GitHub n’utilise que des entiers positifs.
  | access_token | VARCHAR(255) | NOT NULL | Token OAuth pour interroger l’API GitHub au nom de l’utilisateur, chiffré côté application. 255 couvre largement la longueur des tokens GitHub.
  | created_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | Date de création du compte (première connexion OAuth).
### Table t_project
Représente un dépôt GitHub ajouté par un utilisateur pour analyse. La contrainte uk_user_repo empêche le même utilisateur d’ajouter deux fois le même dépôt.
[TABLEAU 6x4]
  | Champ | Type | Contraintes | Rôle et justification
  | pk_project | INT | PK, AUTO_INCREMENT | Identifiant interne du projet.
  | fk_user | INT | NOT NULL, FK → t_user, ON DELETE CASCADE | Propriétaire du projet. Cascade : la suppression du compte supprime ses projets.
  | github_repo_url | VARCHAR(255) | NOT NULL | URL HTTPS complète du dépôt (ex. https://github.com/owner/name). 255 est la taille standard pour une URL.
  | created_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | Date d’ajout du projet à CodeMentor.
  | uk_user_repo | UNIQUE (fk_user, github_repo_url) | Contrainte d’unicité | Évite qu’un même utilisateur ajoute deux fois le même dépôt.
### Table t_analysis
Représente une exécution d’analyse sur un projet. Chaque analyse passe par plusieurs états (queued → running → completed/failed).
[TABLEAU 10x4]
  | Champ | Type | Contraintes | Rôle et justification
  | pk_analysis | INT | PK, AUTO_INCREMENT | Identifiant interne de l’analyse.
  | fk_project | INT | NOT NULL, FK → t_project, ON DELETE CASCADE | Projet analysé. Cascade : la suppression du projet supprime son historique d’analyses.
  | commit_sha | CHAR(40) | Nullable | SHA-1 du commit analysé. CHAR(40) car un SHA-1 fait toujours 40 caractères hexadécimaux : stockage compact et comparaisons rapides.
  | status | ENUM('queued','running','completed','failed') | NOT NULL, DEFAULT 'queued' | État de l’analyse. ENUM choisi car l’ensemble est fermé et connu. Il est stocké sur 1 octet et les valeurs invalides sont rejetées par MySQL.
  | files_total | INT UNSIGNED | Nullable | Nombre de fichiers retenus pour analyse. Nullable car la valeur n’est connue qu’après récupération de l’arborescence.
  | code_overview | TEXT | Nullable | Synthèse globale du projet générée par l’IA. TEXT car la longueur varie fortement.
  | mermaid_diagram | TEXT | Nullable | Code Mermaid.js décrivant la structure du projet. TEXT pour la même raison.
  | error_message | TEXT | Nullable | Message d’erreur détaillé si status = 'failed'.
  | analysis_date | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | Date de lancement de l’analyse.
### Table t_analyzed_file
Représente un fichier traversé par l’IA lors d’une analyse. Chaque fichier porte son propre résumé et ses erreurs détectées.
[TABLEAU 6x4]
  | Champ | Type | Contraintes | Rôle et justification
  | pk_analyzed_file | INT | PK, AUTO_INCREMENT | Identifiant du fichier analysé.
  | fk_analysis | INT | NOT NULL, FK → t_analysis, ON DELETE CASCADE | Analyse à laquelle ce fichier appartient.
  | file_path | VARCHAR(500) | NOT NULL | Chemin relatif du fichier dans le dépôt (ex. src/components/Login.vue). 500 caractères couvrent des arborescences profondes.
  | language | VARCHAR(30) | Nullable | Langage du fichier (javascript, typescript, python...). 30 caractères suffisent largement.
  | file_summary | TEXT | Nullable | Résumé du rôle du fichier généré par l’IA.
### Table t_detected_error
Représente une erreur, un défaut ou une mauvaise pratique détectée par l’IA dans un fichier analysé.
[TABLEAU 10x4]
  | Champ | Type | Contraintes | Rôle et justification
  | pk_detected_error | INT | PK, AUTO_INCREMENT | Identifiant interne de l’erreur.
  | fk_analyzed_file | INT | NOT NULL, FK → t_analyzed_file, ON DELETE CASCADE | Fichier dans lequel l’erreur a été détectée.
  | line_number | INT UNSIGNED | Nullable | Numéro de ligne signalé. Nullable car certaines remarques portent sur le fichier entier.
  | error_type | ENUM('syntax','logic','security','performance','style','deprecation') | NOT NULL | Catégorie d’erreur. ENUM car la liste de catégories est figée par la spécification.
  | severity | ENUM('low','medium','high') | NOT NULL, DEFAULT 'medium' | Niveau de gravité. ENUM car l’ensemble est figé par l’UX (badges colorés).
  | description | TEXT | NOT NULL | Explication détaillée de l’erreur produite par l’IA.
  | code_snippet | TEXT | Nullable | Extrait de code fautif (quelques lignes de contexte).
  | fix_description | TEXT | Nullable | Explication en langage naturel de la correction proposée par l’IA. NULL si aucune correction n’est suggérée (cardinalité 0..1).
  | fix_suggested_code | TEXT | Nullable | Code corrigé proposé par l’IA, prêt à être affiché dans l’interface. NULL si aucune correction n’est suggérée.

## Diagrammes d’interactions
Vue détaillée du scénario principal : lancement d’une analyse et polling de son avancement.
Alt au lieux de loop
Diagramme de séquence : lancement d’une analyse et polling.
Le client envoie POST /projects/:id/analyses au backend.
Le controller enregistre l'analyse en base avec le statut « queued » et la pousse dans la file d'attente en mémoire.
Le controller répond immédiatement 202 avec l'identifiant de l'analyse, sans attendre le traitement.
Le worker récupère l'analyse en file et passe son statut à « running », puis :
Récupère l'arborescence et le contenu du dépôt via l'API GitHub.
Pour chaque fichier, appelle l'API OpenRouter pour générer le résumé et détecter les erreurs.
Met à jour la progression en base après chaque fichier traité.
Génère le diagramme Mermaid en fin de traitement.
Le worker passe le statut à « completed ».
6) En parallèle, le frontend appelle GET /analyses/:id/status toutes les 2 secondes pour mettre à jour la barre de progression.
7) Quand le statut passe à « completed », le frontend appelle GET /analyses/:id pour récupérer le résultat complet (vue d'ensemble, erreurs et corrections, diagramme Mermaid).

### Tests fonctionnels
[TABLEAU 16x4]
  | N° | Objet testé | Description du test | Résultat attendu
  | 1 | Connexion GitHub | L’utilisateur clique sur « Se connecter » et autorise l’application sur GitHub. | L’utilisateur est redirigé vers la liste de ses projets, sa session est active.
  | 2 | Déconnexion | L’utilisateur authentifié clique sur « Se déconnecter ». | La session est fermée et l’utilisateur retourne sur la page de connexion.
  | 3 | Liste des projets | L’utilisateur consulte sa liste de projets. | Ses projets s’affichent avec leur nom, leur langage principal et le nombre d’analyses.
  | 4 | Ajout d’un projet valide | L’utilisateur saisit l’URL d’un dépôt GitHub auquel il a accès. | Le projet est créé et l’utilisateur est redirigé vers la page du projet.
  | 5 | Ajout d’un dépôt inexistant | L’utilisateur saisit une URL pointant vers un dépôt qui n’existe pas. | Un message « dépôt introuvable » est affiché et aucun projet n’est créé.
  | 6 | Ajout d’un dépôt sans accès | L’utilisateur saisit l’URL d’un dépôt privé auquel il n’a pas accès. | Un message « accès refusé » est affiché et aucun projet n’est créé.
  | 7 | Ajout d’un projet en doublon | L’utilisateur ajoute deux fois le même dépôt sur son compte. | Un message « projet déjà existant » est affiché et aucun doublon n’est créé.
  | 8 | Consultation d’un projet | L’utilisateur ouvre la page de détail d’un projet. | Les métadonnées GitHub et l’historique des analyses du projet s’affichent.
  | 9 | Lancement d’une analyse | L’utilisateur clique sur « Lancer une analyse » depuis la page d’un projet. | L’analyse passe au statut « en cours » et l’écran de progression s’affiche.
  | 10 | Suivi de la progression | L’utilisateur reste sur l’écran de progression pendant qu’une analyse est en cours. | La barre de progression se met à jour régulièrement jusqu’à atteindre 100 %.
  | 11 | Consultation des résultats | L’utilisateur ouvre une analyse terminée. | Le diagramme Mermaid, la vue d’ensemble et la liste des erreurs avec leurs corrections suggérées s’affichent.
  | 12 | Échec d’analyse | L’API OpenRouter renvoie une erreur pendant le traitement d’une analyse. | L’analyse passe au statut « échoué » avec un message d’erreur, sans bloquer le worker.
  | 13 | Suppression d’un projet | L’utilisateur supprime un projet qui n’a aucune analyse en cours. | Le projet et son historique d’analyses sont supprimés et la liste est mise à jour.
  | 14 | Suppression bloquée | L’utilisateur tente de supprimer un projet pendant qu’une analyse est en cours. | La suppression est refusée avec un message expliquant qu’une analyse est en cours.
  | 15 | Accès non authentifié | Un visiteur tente d’accéder directement à une page nécessitant une session. | Il est redirigé vers la page de connexion et la ressource n’est pas exposée.
### Tests des services backend (API REST)
Ces tests vérifient directement l’API REST du backend à l’aide de l’interface Swagger. Pour chaque endpoint, on contrôle le cas nominal ainsi que les principaux cas d’erreur, en vérifiant le code HTTP renvoyé et le contenu de la réponse.
[TABLEAU 20x4]
  | N° | Endpoint | Cas testé | Résultat attendu
  | 1 | GET /auth/login | Démarrer la connexion GitHub | 302, redirection vers la page d’autorisation GitHub
  | 2 | GET /auth/me | Demander l’utilisateur courant sans session | 401, accès refusé
  | 3 | GET /auth/me | Demander l’utilisateur courant avec une session valide | 200, données de l’utilisateur (le token GitHub n’est pas renvoyé)
  | 4 | POST /auth/logout | Se déconnecter avec une session active | 204, la session est fermée
  | 5 | GET /projects | Lister ses projets sans session | 401, accès refusé
  | 6 | GET /projects | Lister ses projets avec une session valide | 200, liste des projets de l’utilisateur
  | 7 | POST /projects | Ajouter un dépôt GitHub valide et accessible | 201, le projet est créé
  | 8 | POST /projects | Envoyer une URL qui n’est pas un dépôt GitHub | 400, données invalides
  | 9 | POST /projects | Ajouter un dépôt qui n’existe pas | 404, dépôt introuvable
  | 10 | POST /projects | Ajouter un dépôt privé sans y avoir accès | 403, accès refusé
  | 11 | POST /projects | Ajouter deux fois le même dépôt | 409, projet déjà existant
  | 12 | GET /projects/:id | Consulter un de ses projets | 200, détail du projet et historique des analyses
  | 13 | GET /projects/:id | Consulter un projet d’un autre utilisateur ou inexistant | 404, ressource introuvable
  | 14 | DELETE /projects/:id | Supprimer un de ses projets sans analyse en cours | 204, le projet et son historique sont supprimés
  | 15 | DELETE /projects/:id | Supprimer un projet pendant qu’une analyse est en cours | 409, suppression refusée
  | 16 | POST /projects/:id/analyses | Lancer une analyse sur un de ses projets | 202, analyse créée au statut « queued », son identifiant est renvoyé
  | 17 | GET /analyses/:id/status | Suivre la progression d’une analyse en cours | 200, statut et pourcentage d’avancement
  | 18 | GET /analyses/:id | Consulter le résultat d’une analyse terminée | 200, vue d’ensemble, fichiers, erreurs, corrections et diagramme Mermaid
  | 19 | GET /analyses/:id | Consulter une analyse d’un autre utilisateur ou inexistante | 404, ressource introuvable
